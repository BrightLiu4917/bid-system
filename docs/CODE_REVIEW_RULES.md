# 代码审查规则

## 审查优先级
1. 业务正确性
2. 数据安全
3. 安全性
4. 权限和租户
5. 事务安全
6. API 和前端兼容
7. 测试和验证
8. 性能
9. 可维护性
10. 代码风格

## 严重级别

### P0 阻塞
必须阻止合入或发布的问题。包括数据破坏、越权、租户隔离缺失、未确认业务规则、编译失败、测试关键路径失败、破坏 API 兼容、危险 SQL、明文密钥泄露、未经确认的 DROP/TRUNCATE/全表 UPDATE/DELETE。

### P1 重要
应在本次合入前修复，除非有明确豁免和补救计划。包括事务边界缺失、错误态缺失、N+1 风险、索引缺失、分页不可控、重复提交风险、审计日志缺失、关键测试缺口、文档与实现不一致。

### P2 次要
不阻塞发布但应记录的问题。包括命名不清、重复代码、局部可读性问题、非关键文案不一致、低风险边界测试缺失、可维护性改进建议。

## 审查方法
- 先看 OpenSpec change/spec，再看 diff，最后看调用方和验证证据。
- 问题必须可定位到文件和最小范围。
- 每个问题说明影响、触发条件和修复方向。
- 不把个人偏好写成阻塞问题。
- 证据不足时列为待确认问题，不猜测。
- 发现无关修改时要求拆分或说明原因。

## 阻塞问题
出现以下情况必须拒绝变更：
- 代码无法编译。
- 业务规则是猜测出来的。
- 状态流转错误。
- 权限或租户隔离缺失。
- 多表写入缺少事务。
- SQL 可能破坏数据。
- 表结构变化缺少 migration SQL。
- API 响应结构破坏兼容性。
- 前端缺失关键状态处理。
- 暴露敏感数据。
- 引入未经批准的依赖。
- 修改无关文件。
- 验证步骤缺失且变更影响生产行为。
- OpenSpec 与实现冲突。

## 输出格式
1. 阻塞问题
2. 重要问题
3. 次要建议
4. 表结构/字段变化审查
5. Entity/Mapper/XML/DTO/VO 与表结构一致性审查
6. API/前端契约审查
7. SQL 性能风险审查
8. 安全和权限审查
9. 验证步骤
10. 风险总结
