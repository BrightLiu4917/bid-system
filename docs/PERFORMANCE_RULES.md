# 性能规则

## 后端
- 避免 N+1 查询。
- 避免大表无索引扫描。
- 分页查询必须可控。
- 高基数字段和时间范围查询应考虑索引。
- 大批量操作必须说明影响范围和分批策略。

## N+1 检测
- 阅读列表页、详情聚合和导出接口的循环查询。
- 检查 Service 中是否在 `for` / `stream` / `map` 内调用 Mapper、Repository、远程接口或缓存。
- 对分页列表必须说明主查询、关联数据加载方式和查询次数上限。
- 需要补充日志或测试时，应记录 SQL 数量、参数规模和触发路径。

## 大表分页
- 大表列表禁止无限滚动式全量加载。
- 深分页风险高时优先考虑游标分页、基于稳定排序键的 seek pagination 或限制最大页数。
- 排序字段必须稳定，常见组合为业务时间、主键或唯一序列。
- 筛选条件、排序字段和租户/软删除字段应纳入索引设计。
- 导出任务应考虑异步化、分批读取和结果文件生命周期。

## 缓存策略
- 缓存必须说明 key、过期时间、失效时机和数据一致性要求。
- 禁止用缓存掩盖错误查询或缺失索引。
- 权限、租户、用户维度数据必须进入缓存 key 或明确隔离。
- 高并发热点缓存应考虑穿透、击穿和雪崩保护。
- 写入后读取一致性要求高的场景必须说明是否允许短暂旧值。

## 前端
- 避免不必要的全量渲染。
- 大列表应分页、虚拟滚动或按需加载。
- 避免重复请求。
- 加载态和错误态必须明确。

## 输出要求
涉及性能风险时必须说明：
- 数据规模假设
- 查询路径
- 索引使用
- 缓存策略，如有
- 限流或分页策略
- N+1 检测结论
- 大表、导出或批处理策略
- 验证方式
