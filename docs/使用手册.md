# AI 全栈控制系统使用手册

## 这是什么

AI 全栈控制系统是一套复制到业务项目根目录使用的工程约束和协作流程。

它不是业务框架，也不是运行时服务。它的作用是把 AI 开发拆成可确认、可审查、可验证的步骤：

```text
需求事实源 -> 架构设计 -> 数据库/API/任务拆分 -> 本地模型写代码 -> 测试 -> 二审 -> Codex 总审
```

## 角色分工

| 角色 | 负责什么 | 不负责什么 |
|------|----------|------------|
| Codex | 需求澄清、系统架构、表设计、任务拆分、代码审查、风险控制 | 不在规则不清时猜业务 |
| OpenSpec | 记录已确认需求、设计、任务、规格 | 不替代用户确认 |
| OpenCode + qwen-coder | 按任务文件执行本地代码修改 | 不决定字段、状态、权限、API |
| deepv4 | 对 diff、测试结果、风险点做二审 | 不直接改代码 |
| scripts | 安装、检查、测试、日志摘要 | 不保证业务正确性 |

## 标准开发流程

### 1. 先写事实源

新需求不要直接让 AI 改代码。先创建：

```text
openspec/changes/<change-id>/proposal.md
openspec/changes/<change-id>/design.md
openspec/changes/<change-id>/tasks.md
openspec/changes/<change-id>/specs/<capability>/spec.md
```

可以从模板复制：

```text
templates/openspec-change/
```

### 2. Codex 做架构和拆分

让 Codex 读取 `AGENTS.md`、`CONTEXT.md` 和对应 OpenSpec change，然后产出：

- 需求理解
- 功能范围
- 表结构影响
- API 影响
- 前后端任务拆分
- 测试计划
- 风险和待确认问题

### 3. 本地 Agent 写代码

如果启用 OpenCode/qwen-coder，Codex 会生成：

```text
local-agent-task.md
```

里面必须包含：

- 允许修改范围
- 禁止修改范围
- 需要遵守的 skill 规则摘要
- 测试命令
- 停止条件

### 4. 执行测试

统一测试入口：

```bash
bash scripts/run-tests.sh
```

测试命令来自：

```text
.agent/local-agent.env
PROJECT_TEST_COMMAND='...'
```

### 5. deepv4 二审

如启用 deepv4：

```bash
bash scripts/prepare-deepv4-review.sh
bash scripts/run-deepv4-review.sh
```

deepv4 只看审查输入，不直接修改代码。

### 6. Codex 最终审查

Codex 最终检查：

- 是否符合 OpenSpec
- 是否有未确认字段、状态、权限、API
- 是否有 SQL 风险
- 是否有事务问题
- 是否有前后端契约不一致
- 是否有测试缺口

## 常用命令

```bash
bash scripts/setup-control-system.sh
bash scripts/bootstrap-new-project.sh --project-dir /path/to/project --profile local-agent
bash scripts/skill-check.sh
bash scripts/route-check.sh
bash scripts/docs-link-check.sh
bash scripts/openspec-check.sh openspec/changes/<change-id>
bash scripts/openspec-conflict-check.sh
bash scripts/run-tests.sh
```

## 使用原则

- 不确定的业务规则必须先问用户。
- 数据库、API、权限、状态流转必须进入 OpenSpec。
- qwen-coder 只执行任务，不决定业务。
- deepv4 只做二审，不替代 Codex 和用户确认。
- 代码实现必须可测试、可审查、可回滚。
