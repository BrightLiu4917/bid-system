# UX 规则

## 必须考虑
- 用户进入页面的入口。
- 用户完成任务的主路径。
- 空态、加载态、错误态、成功态。
- 权限不足、数据不存在、网络失败。
- 表单校验、错误提示和提交反馈。
- 批量操作和高风险操作确认。
- 移动端和桌面端可用性，如项目需要。

## 状态设计
- 加载态必须说明触发时机、持续期间可执行动作和超时反馈。
- 空态必须区分“无权限查看”“有权限但无数据”“筛选条件无结果”。
- 错误态必须提供可执行恢复路径，例如重试、返回、修改输入或联系管理员。
- 成功态必须说明下一步去向，避免用户不知道操作是否生效。
- 权限态必须避免暴露用户无权操作的数据细节。

## 表单和批量操作
- 表单字段必须明确必填、格式、范围、默认值和禁用条件。
- 提交中必须防重复提交。
- 批量操作必须说明选择范围、跨页选择语义、失败项展示和部分成功处理。
- 高风险操作必须二次确认，确认文案应包含对象、影响和不可逆风险。
- 删除、作废、扣减、发布、回滚等动作必须说明撤销或补救路径。

## 导航和信息架构
- 页面入口必须可从既有导航或业务流程自然到达。
- 列表、详情、编辑、创建之间的返回路径必须明确。
- 筛选、排序、分页和刷新不能让用户丢失关键上下文。
- 多角色页面必须说明不同角色可见内容和可执行动作。

## 禁止事项
- 禁止用说明文字代替清晰交互。
- 禁止缺失失败反馈。
- 禁止高风险操作无确认。
- 禁止在用户未确认时自动执行破坏性动作。

## 输出要求
涉及 UI/UX 时必须说明：
- 页面或流程入口
- 用户主流程
- 异常流程
- 状态设计
- 权限态设计
- 表单和校验
- 批量操作和高风险确认，如有
- 返回、刷新和重复提交策略
- 待确认交互
